# Project Overview — Product & Design Requirements

## Purpose

alpha-signal-reminder solves a specific, painful problem for solo crypto traders: token address signals posted in Discord channels and Telegram chats are worthless if you miss them by five minutes. Price impact on new token launches can move 10x–100x within seconds of a signal appearing. Checking a phone screen manually is not viable at 3 AM.

This project creates a 24/7 personal alert daemon. When a token address signal appears in a monitored channel, it rings your iPhone immediately — no app open, no notification silenced, no missed call.

---

## Target User

A solo trader or developer who:

- Monitors 1–5 Discord servers and Telegram channels for alpha signals (token contract addresses)
- Trades on EVM chains (Ethereum, BSC, Base, Arbitrum) and/or Solana
- Wants a phone call, not a push notification — calls are harder to sleep through
- Runs a personal server or VPS capable of running Node.js with PM2
- Is comfortable with a one-time `.env` configuration and `npm install`

This is not a product for teams. It is a personal tool. No dashboard, no multi-user auth, no paid subscription.

---

## Core Features

- **Discord monitoring** — connects a bot to any Discord server and watches for new messages; triggers a call on every message (filtered by channel if configured)
- **Telegram monitoring** — connects as a user account (MTProto via GramJS) to listen to channels and groups; only triggers a call when a token contract address is detected in the message text
- **EVM token address detection** — regex match on `0x` followed by exactly 40 hex characters
- **Solana token address detection** — regex match on base58 strings of 43–44 characters
- **SIP calling (free)** — sends a SIP INVITE directly to `sip.linphone.org`, which uses PushKit to wake the Linphone iOS app and ring the phone even when the app is closed
- **Twilio PSTN calling (paid)** — places a real phone call using TwiML voice synthesis as a fallback or alternative
- **Per-source cooldown** — configurable quiet period (default 5 minutes) per source to prevent a burst of signals from placing ten calls in a row
- **HTTP webhook** — `GET /webhook` manually triggers a call for testing or integration with external tools
- **SIP test endpoint** — `GET /test-sip` tests the SIP call path directly without going through the full alert pipeline
- **Health endpoint** — `GET /health` returns uptime and server status for process monitoring
- **PM2 integration** — single-instance process manager config with autorestart and memory cap

## What Triggers a Call

| Source | Trigger condition |
|---|---|
| Discord | Any new message in a monitored channel (after cooldown) |
| Telegram | A new message containing at least one EVM or Solana token address (after cooldown) |
| HTTP webhook | `GET /webhook` request |

The Discord bot triggers on any message because Discord alpha channels are typically curated — every message is intentional. Telegram triggers only on token address detection because Telegram groups are often noisy and include commentary, reply chains, and off-topic discussion.

## Alert Lifecycle

```
Signal arrives (Discord message or Telegram token address detected)
    |
    v
Cooldown check — if in cooldown window: log and skip
    |
    v
Cooldown reset — record timestamp
    |
    v
makeCall() — dispatch to active provider
    |
    +--[CALL_PROVIDER=sip]--> SIP INVITE to sip.linphone.org
    |                              --> push notification to Linphone iOS
    |                              --> iPhone rings (CallKit UI)
    |
    +--[CALL_PROVIDER=twilio]--> Twilio REST API
                                   --> PSTN call to CALL_TO_NUMBER
                                   --> phone rings
```

A call result of `success: true` means the alert was dispatched. For SIP, this includes the case where the user declines the call (603 Decline) — the phone rang, which was the goal. For Twilio, success means the call was accepted by the Twilio API and a call SID was returned.

## Supported Call Providers

### SIP (free, default)

Uses the `sip` npm package to craft a raw SIP INVITE message. No SIP account registration is required. The bot sends from a fictional identity (`alertbot@iptel.org`) to the target Linphone account (`youruser@sip.linphone.org`). The `sip.linphone.org` server accepts the INVITE as a federated call and delivers it via Apple PushKit.

See [system-architecture.md](./system-architecture.md) for the full SIP call flow.

### Twilio (paid)

Uses the Twilio REST API to place a PSTN call. The call plays a short synthesized voice message generated by TwiML. Trial accounts are restricted to verified phone numbers only. Per-minute pricing applies.

## Non-Goals

This project intentionally does not:

- **Execute trades** — it alerts, it does not buy or sell tokens
- **Analyze signals** — it detects a token address pattern, it does not evaluate whether the signal is credible or profitable
- **Support multiple users** — there is no authentication layer, no user database, no per-user configuration
- **Run as a SaaS** — there is no hosted version, no billing system, no onboarding flow
- **Aggregate or store signals** — detected addresses are logged but not persisted to a database or searchable index
- **Send notifications via SMS, email, or push** — the only notification channel is a phone call
- **Support chains beyond EVM and Solana** — other address formats are not detected
- **Operate as a public Discord bot** — the bot is private and must be added to your own server

## Future Enhancement Ideas

These are not planned work, but logical next steps if the tool's scope were to expand:

- **Address deduplication** — skip a call if the exact same address was already seen within a configurable window
- **Signal source tagging** — include the source channel name in the TwiML or SIP caller ID
- **Multi-provider fallback** — try SIP first, fall back to Twilio if SIP fails
- **Solana address length expansion** — Solana addresses can be 32–44 characters; the current regex uses 43–44 to reduce false positives, but this could be tuned
- **Rate limiting the webhook endpoint** — the HTTP endpoint currently has no authentication; IP-based rate limiting would reduce abuse risk
- **Telegram group filtering** — currently filters by a single channel ID; could support a list of IDs
- **Discord keyword filter** — trigger only when specific keywords (e.g., "new contract", "launch") appear alongside a token address
- **Startup notification** — send a test call or log message on first boot so you know the daemon is running correctly
